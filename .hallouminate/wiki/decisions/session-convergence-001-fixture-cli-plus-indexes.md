# ADR session-convergence-001: Analytics DB stays CLI-queried — no MCP, no ingest timer, no secondary indexes  [status: accepted]

- **Context:** The user asked for the session-analytics duckdb (~730 MB, `~/.claude/analytics/sessions.duckdb`) to "maybe become a permanent fixture for llms to query". Candidates: a read-only duckdb MCP, a systemd ingest timer, both, or the status quo plus indexing. The host livelocked 2026-07-08 on standing I/O load (see [[operations/local-llm]]), and a timed `--force` ingest measured 30 s wall, ~1 GB read / ~1.8 GB written, 2.4 GB peak RSS per full rebuild (`ingest.py` is a full re-walk + full-table rebuild each run — no incremental path).
- **Decision:** CLI-only query surface (duckdb-expert sub-agent, `~/.duckdbrc` 8 GB cap). Lazy 1h-TTL refresh stays; no MCP server, no standing timer. **No secondary indexes** — see the superseded sub-decision below.
- **Alternatives:** Read-only MCP + on-demand ingest (rejected by user — new surface not justified); MCP + systemd timer (rejected — a recurring full-rebuild ingest is exactly the standing I/O shape the box was just hardened against).
- **Superseded sub-decision (2026-07-09) — secondary indexes do NOT help on DuckDB.** The original ADR paired the CLI surface with "proper profile-derived indexes". Implementation disproved that half: `CREATE INDEX` on the analytics-pack predicates was added, then **reverted** after `EXPLAIN` on the live DB showed `SEQ_SCAN` for every indexed predicate — including a selective `sessionId` point-lookup on 273k-row `raw_entries`, and even the pre-existing indexes. **DuckDB's columnar engine does not use secondary ART indexes for base-table filter scans** (they exist mainly for constraint enforcement, not analytical scans). Secondary indexes on this DB are dead weight. The real perf levers are the automatic columnar zonemaps (min/max per row-group), `ORDER BY` / clustering at ingest, or materialized rollups — not indexes. Full-scan query cost is already fine (ingest ~10.6 s; pack queries scan the 720 MB DB fast).
- **Consequences:** Zero new standing processes and zero secondary indexes; queries pay the ≤30 s ingest cost when TTL lapses. If the DB later needs to be genuinely always-fresh, ingest must first grow an incremental path — revisit this ADR then. If pack-query latency ever becomes a real problem, reach for clustering/rollups, not `CREATE INDEX`. Rejected directions recorded in `.cheese/.out-of-scope/session-convergence-002.md`.
