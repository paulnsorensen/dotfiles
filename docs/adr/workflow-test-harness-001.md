### ADR-001: Execute real workflow-script bodies under a faithful mini-runtime  [status: accepted]

- **Context:** `tests/workflows-parse.sh` guarded `claude/workflows/*.js` with parse checks, copy-pasted re-implementations of script-internal functions, and `src.includes()` string greps — tests could pass while the shipped script's behavior regressed. Dynamic-workflow scripts are plain sandboxed JS with injected globals (`agent/pipeline/parallel/phase/log/workflow/args/budget`), so the bodies are executable offline.
- **Decision:** A test harness (`tests/workflows/harness.mjs`) strips the `meta` export, compiles the body as a `vm.Script` run in a `vm.createContext` sandbox with `codeGeneration: { strings: false, wasm: false }`, and executes it against a mini-runtime that reproduces documented semantics: no-barrier `pipeline` with null-drop on stage throw, null-on-throw `parallel` barrier, scriptable `budget`, `Date`/`Math` proxies that throw (sandbox parity), and per-call validation of stub fixtures against the declared `agent()` schema.
- **Alternatives:** (a) Keep grep-based static checks — rejected: exactly the false confidence being removed. (b) Minimal stubs (naive map/Promise.all, no schema validation) — rejected: tests would pass where the real runtime nulls an item or a schema drifted.
- **Consequences:** Control-flow regressions (unbounded loops, wrong routing, broken arg coercion) and schema drift fail tests before deploy. Cost: the mini-runtime must track upstream runtime semantics (research preview); each behavior is source-annotated in the harness header to keep that diffable.
