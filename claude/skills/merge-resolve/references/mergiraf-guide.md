# Mergiraf Reference Guide

AST-aware structural merge driver for git. Parses source code via Tree-sitter,
computes structural diffs, and merges at the syntax tree level.

## CLI Reference

### mergiraf merge

The main merge driver, invoked by git automatically.

```
mergiraf merge --git <ancestor> <ours> <theirs> [flags]
```

| Flag | Git Variable | Purpose |
|------|-------------|---------|
| `--git` | â€” | Enable git mode (write result to `<ours>` in place) |
| `-s` | `%S` | File size in bytes (skip structural merge above threshold) |
| `-x` | `%X` | Label for "ours" in conflict markers |
| `-y` | `%Y` | Label for "theirs" in conflict markers |
| `-p` | `%P` | File path (used to detect language from extension) |

**Exit codes:**
- `0` â€” Clean merge, no conflicts
- `1` â€” Conflicts remain (written as standard `<<<<<<<` markers)

### mergiraf merge (standalone, non-git mode)

Three-way merge without git integration. Use `-o` for output file, `-p` for
language detection from path:

```
mergiraf merge <base> <left> <right> -o <output> -p <path-for-language>
```

Check for conflict markers in output to determine if resolution is clean:
```bash
mergiraf merge base ours theirs -o merged -p file.py
grep -c '<<<<<<' merged  # 0 = clean, >0 = conflicts remain
```

### mergiraf solve

Resolves conflicts in a file that already has conflict markers (post-merge).
Parses the conflict markers, reconstructs the three-way inputs, and attempts
structural resolution.

```
mergiraf solve <file-with-conflicts> [options]
```

| Flag | Purpose |
|------|---------|
| `-p` / `--stdout` | Print result to stdout instead of overwriting the file |
| `--keep-backup` | Keep `.orig` backup of the original (default: true) |
| `-L` / `--language` | Override automatic language detection |
| `-v` / `--verbose` | Verbose output |

```bash
# Preview resolution without modifying the file
mergiraf solve path/to/conflicted-file.py --stdout

# Apply resolution in place
mergiraf solve path/to/conflicted-file.py
```

### mergiraf languages

List supported languages and their file extensions.

```
mergiraf languages                  # Human-readable table
mergiraf languages --gitattributes  # gitattributes format (for ~/.gitattributes)
```

### Environment Variables

| Variable | Purpose |
|----------|---------|
| `RUST_LOG=mergiraf=debug` | Show parse results, matching decisions, fallback reasons |
| `RUST_LOG=mergiraf=trace` | Verbose: full tree-sitter parse output |

```bash
RUST_LOG=mergiraf=debug mergiraf merge base ours theirs -o merged -p file.py 2>&1
```

Debug output goes to stderr; merged content to the output file.

## How It Works

### 1. Parsing Phase

All three file versions (ancestor, ours, theirs) are parsed into concrete syntax
trees via Tree-sitter. Mergiraf bundles its own parsers â€” no external grammar
installation required.

### 2. Node Matching

A matching algorithm identifies corresponding nodes across the three trees:
- **Exact hash matching** (fast path) for identical subtrees
- **Similarity scoring** for modified nodes (GumTree-style tree differencing)

### 3. Structural 3-Way Merge

With the matching established, mergiraf classifies each change:

| Change Type | Resolution |
|------------|------------|
| Independent additions at different positions | Both included (auto-merged) |
| One side changed a node, other left it alone | Take the changed version |
| Both sides modified the same node differently | Conflict markers |
| Order-insensitive constructs (imports, use) | Merged without position conflicts |

### 4. Fallback

If mergiraf can't parse a file or structural merge yields more conflicts than
text merge, it falls back to standard git 3-way merge. The result is never
worse than what git's built-in merge would produce.

### What Structural Merge Resolves That Text Merge Can't

- Two devs each add an `import`/`use` at the top of the file in different positions
- Two devs each add a new function to a class in different positions
- One dev adds a struct field, another modifies a different field
- JSON/YAML: independent key additions at the same level
- `go.mod`/`go.sum`: dependency additions from concurrent feature branches

## Supported Languages

Generated by `mergiraf languages --gitattributes`. 77 file extensions across 40+ languages:

**Systems**: Rust (`.rs`), C (`.c`, `.h`), C++ (`.cc`, `.hh`, `.cpp`, `.hpp`, `.cxx`,
`.hxx`, `.c++`, `.h++`, `.mpp`, `.cppm`, `.ixx`, `.tcc`), C# (`.cs`), Go (`.go`,
`go.mod`, `go.sum`, `go.work.sum`), SystemVerilog (`.sv`, `.svh`)

**JVM**: Java (`.java`, `.properties`), Kotlin (`.kt`), Scala (`.scala`, `.sbt`)

**Web**: JavaScript (`.js`, `.jsx`, `.mjs`), TypeScript (`.ts`, `.tsx`),
HTML (`.html`, `.htm`, `.xhtml`), PHP (`.php`, `.phtml`, `.php3`-`.php5`, `.phps`, `.phpt`)

**Scripting**: Python (`.py`), Ruby (`.rb`), Lua (`.lua`), Elixir (`.ex`, `.exs`)

**Data/Config**: JSON (`.json`), YAML (`.yml`, `.yaml`), TOML (`.toml`, `pyproject.toml`),
XML (`.xml`), INI (`.ini`), Properties (`.properties`)

**Infrastructure**: HCL/Terraform (`.hcl`, `.tf`, `.tfvars`), Nix (`.nix`),
Makefile (`Makefile`, `GNUmakefile`, `.mk`), CMake (`CMakeLists.txt`, `.cmake`),
Bazel/Starlark (`.bzl`, `.bazel`, `BUILD`, `WORKSPACE`)

**Other**: Dart (`.dart`), Solidity (`.sol`), OCaml (`.ml`, `.mli`),
Haskell (`.hs`), Markdown (`.md`), Device Tree Source (`.dts`)

**Not supported** (falls back to text merge): Shell (`.sh`), Swift, SQL, R, Perl, Julia

## Git Configuration

### Merge Driver (gitconfig)

```ini
[merge "mergiraf"]
    name = mergiraf
    driver = mergiraf merge --git "%O" "%A" "%B" -s "%S" -x "%X" -y "%Y" -p "%P"
```

The `-p "%P"` flag is critical â€” mergiraf uses the file path extension to select
the correct Tree-sitter grammar.

### File Registration (gitattributes)

Global (applies to all repos):
```bash
mergiraf languages --gitattributes > ~/.gitattributes
# Requires: core.attributesfile = ~/.gitattributes in gitconfig
```

Per-repo (overrides global):
```bash
mergiraf languages --gitattributes > .gitattributes
```

### Disabling for Specific Files

In a repo's `.gitattributes`:
```
*.py merge=            # Reset to text merge (overrides global)
package-lock.json merge=ours   # Always keep ours for lockfiles
```

### Upgrading

After `brew upgrade mergiraf`, regenerate gitattributes to pick up newly
supported languages:
```bash
mergiraf languages --gitattributes > ~/.gitattributes
```

## Resolution Patterns

### Pattern 1: Parallel Feature Branches with Shared Files

Two features both touch `config.yaml` or a shared `types.ts`:
```bash
git merge feature-b
# mergiraf: both sets of changes included, no conflict
```

### Pattern 2: Long-Running Rebase

```bash
git rebase main
# Each commit: mergiraf resolves structural conflicts silently
# rerere replays any previously manually resolved conflicts
# Net: drastically fewer interactive resolution sessions
```

### Pattern 3: Cherry-Pick Workflow

```bash
git cherry-pick sha1 sha2 sha3
# mergiraf runs on each file per cherry-pick
# If conflicts remain: git stops, fix manually, git cherry-pick --continue
```

### Pattern 4: Preview Before Merge

```bash
git show MERGE_HEAD:path/file.rs > /tmp/theirs.rs
git show HEAD:path/file.rs > /tmp/ours.rs
git show $(git merge-base HEAD MERGE_HEAD):path/file.rs > /tmp/base.rs
mergiraf merge /tmp/base.rs /tmp/ours.rs /tmp/theirs.rs -o /tmp/merged.rs -p file.rs
cat /tmp/merged.rs
```

### Pattern 5: Audit Mergiraf Behavior

```bash
RUST_LOG=mergiraf=debug git merge --no-ff feature-branch 2>&1 | grep mergiraf
```

## Integration

### mergiraf + rerere

- mergiraf runs first as the merge driver, before rerere sees anything
- Clean structural merge â†’ rerere records nothing (nothing to record)
- Conflict markers remain â†’ rerere checks its database for a matching resolution
- The combination: first occurrence needs manual resolution once; all subsequent
  cherry-picks/rebases with the same conflict are auto-resolved

### mergiraf + kdiff3

- They never run simultaneously â€” mergiraf is the driver (during `git merge`),
  kdiff3 is the mergetool (after, via `git mergetool`)
- kdiff3 sees mergiraf's partially-resolved output â€” fewer conflicts to handle
  than a raw 3-way merge

### mergiraf + zdiff3

The `conflictstyle = zdiff3` setting shows ancestor context in conflict markers.
Works fine with mergiraf output â€” git post-processes conflict markers through
the configured style.

## Troubleshooting

### File Not Getting Structural Merge

```bash
# Check if the extension is registered
git check-attr merge -- path/to/file.ext
# Expected: path/to/file.ext: merge: mergiraf

# If not, check gitattributes
cat ~/.gitattributes | grep '<extension>'

# Regenerate if missing
mergiraf languages --gitattributes > ~/.gitattributes
```

### Parse Failures (Silent Fallback)

If a file has a syntax error in any of the three versions, Tree-sitter may
fail to produce a complete parse tree. Mergiraf falls back to text merge
silently. Detect with:

```bash
RUST_LOG=mergiraf=debug mergiraf merge base ours theirs -o merged -p file.py 2>&1 | grep -i "parse\|fallback"
```

### Whitespace-Only Formatting Changes

If one side reformatted (rustfmt, prettier) while the other modified content,
mergiraf may produce more conflicts because AST positions shifted.

**Resolution**: run the formatter on both branches before merging, or reformat
the merged result after resolution.

### Lockfile Handling

| Lockfile | Recommendation |
|----------|---------------|
| `Cargo.lock` | mergiraf merges as TOML â€” usually correct. Verify: `cargo check` |
| `package-lock.json` | Regenerate: `npm install` after resolving `package.json` |
| `yarn.lock` | Regenerate: `yarn install` |
| `pnpm-lock.yaml` | Regenerate: `pnpm install` |

Consider adding to repo `.gitattributes`:
```
package-lock.json merge=ours
yarn.lock merge=ours
pnpm-lock.yaml merge=ours
```

### rerere Replaying a Stale Resolution

When rerere replays a resolution from a text-merged conflict but mergiraf
has restructured the file differently, rerere's resolution may not apply
cleanly. If rerere fails, you get the mergiraf-with-conflict-markers state
and must resolve again manually.

Fix: `git rerere forget <path>` to clear the stale resolution.

### Import Ordering After Merge

Mergiraf may reorder imports when merging two sets of additions, using
AST-level canonical ordering. The output compiles/runs correctly but may
differ from what either branch had. Run your linter/formatter after merge
if you have strict import ordering rules.

## Performance

- Files under ~5,000 lines: fast (sub-second)
- Very large generated files (minified JS, protobuf-generated): noticeably slower
- The `-s %S` flag passes file size â€” mergiraf skips structural merge above ~1MB
- Tree-sitter parsing is O(n); node matching is O(n log n) in tree size
- Custom size limit: `mergiraf merge --size-limit 500000` (bytes)

### Large Repo Tips

- Disable for generated code: `path/to/generated/ merge=` in `.gitattributes`
- Disable for lockfiles that prefer overwrite: `package-lock.json merge=ours`
- Team-shared `.gitattributes` is safe â€” git falls back to text merge if mergiraf
  binary isn't installed on a contributor's machine
